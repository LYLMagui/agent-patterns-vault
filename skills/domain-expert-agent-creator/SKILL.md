# Domain Expert Agent Creator

## 目标

把“你是一个 XX 专家”的模糊请求，升级为可执行、可约束、可评测、可复用的专家级 Agent Prompt。

这个 skill 必须默认自包含可用，不能假设用户的工作区里已经有任何现成条目、模板目录或仓库结构。

优先生成两层结果：

1. 可直接保存或分享的通用角色条目
2. 可直接复制到模型中的运行时 Prompt

默认使用下面这套通用结构，不依赖任何特定仓库格式：

- `Role`
- `Persona`
- `Core Principles`
- `Decision Framework`
- `Communication Style`
- `Boundaries`
- `Tool / Skill Routing`
- `Output Format`

## 输入资料规则

默认直接基于用户当前提供的信息生成结果，不依赖任何预置参考资料。

如果使用者有以下内容，应由使用者主动提供，而不是由 skill 假设其存在：

- 既有角色模板
- 组织内部提示词规范
- 指定的目录结构或命名规范
- 某位人物、行业或方法论的参考材料

如果这些资料没有提供，就使用本 skill 内置的通用结构完成生成，不要停止。

## 步骤 1：先判断是否真的适合做成“单一专家 Agent”

先把用户需求归类，不要一上来就堆设定。

- `领域专家`：如架构师、税务顾问、增长负责人、SRE
- `具名人物风格专家`：如“Bezos 风格 CEO”“Norman 风格设计顾问”
- `虚构角色`：如世界观内的顾问、导师、批评者
- `批判型角色`：如反方审稿人、风险官、质控官
- `执行型角色`：如运营经理、交付 PM、研究分析师

如果用户把过多职责塞进一个角色，先收缩范围。

- 同时要求“研究 + 决策 + 执行 + 审计 + 长期记忆”的，优先拆成多个角色
- 高复杂度任务优先使用“主专家 + 审稿/研究/执行配角”模式
- 不要为了显得强大而制造超载 Prompt

## 步骤 2：抽取生成该专家所需的最小输入

至少明确以下内容。缺失时可以做最保守假设，但必须在结果里显式标出假设项。

- 专家名称：角色名、岗位名、人物名
- 适用领域：行业、学科、业务场景、技术栈
- 核心目标：要帮用户完成什么
- 关键任务：诊断、规划、评审、生成、决策、执行中的哪些
- 非目标：该专家不负责什么
- 目标用户：创始人、工程师、运营、普通用户、研究者等
- 证据标准：是否必须引用来源、数据、法规、文献或原文
- 风险等级：普通任务 / 高风险任务（医疗、法律、金融、安全等）
- 可用工具：MCP、CLI、浏览器、文件处理、数据库、搜索
- 可用 skills：本地已有 skill，或需要额外安装的 skill
- 输出要求：语气、长度、结构、交付格式

## 步骤 3：构建角色蓝图，而不是只写“人设”

按下面的层次构建角色，避免空洞设定。

### 3.1 身份层

- 该角色是谁
- 他为什么可信
- 他的职责边界是什么

### 3.2 认知层

- 该角色默认使用哪些思维模型
- 他优先看重哪些指标、原则、取舍
- 他常用什么问题框架

### 3.3 知识边界层

- 他熟悉什么
- 他不负责什么
- 哪些内容必须查证后再回答

### 3.4 行为层

- 遇到信息不足时怎么做
- 遇到高风险建议时怎么做
- 遇到用户需求越界时怎么拒绝或转交

### 3.5 交流层

- 语气与风格
- 输出密度
- 是否直接给结论、是否保留备选方案

## 步骤 4：优先使用结构化 Prompt，而不是散文式说明

Prompt 变长、模块变多时，优先使用 XML 标签或明确的 Markdown 模块。这样更稳定，也更利于后续维护。

### 推荐模块

必须覆盖以下模块：

1. `Role`
2. `Profile`
3. `Goals`
4. `Non-Goals`
5. `Background`
6. `Knowledge Boundaries`
7. `Constraints`
8. `Tools and Skills`
9. `Workflow`
10. `Output Contract`
11. `Context / Memory Rules`
12. `Self-Check`
13. `Few-Shot Examples`
14. `Initial Message`

### 角色条目模板

当需要产出一个可保存、可分享、可迁移的角色条目时，优先按这个结构输出：

```markdown
# [领域/岗位] Agent - [角色名]

## Role
[一句话说明该角色负责什么]

## Persona
[说明其背景、视角、思维方式、默认立场]

## Core Principles
### [原则 1]
- [具体原则]
- [具体原则]

### [原则 2]
- [具体原则]
- [具体原则]

## Decision Framework
### 当需要分析问题时
1. [分析问题 1]
2. [分析问题 2]
3. [分析问题 3]

### 当需要做取舍时
1. [取舍问题 1]
2. [取舍问题 2]
3. [取舍问题 3]

## Communication Style
- [风格]
- [密度]
- [偏好]

## Boundaries
- [明确不做什么]
- [需要证据时的处理方式]
- [高风险话题的处理方式]

## Tool / Skill Routing
- 当任务需要额外能力时，先检查现有 skills
- 若本地缺少所需能力，显式使用 `find-skills` 搜索相关 skill
- 对每类任务说明该调用什么能力

## Output Format
1. [先输出什么]
2. [再输出什么]
3. [最后输出什么]
```

### 运行时 Prompt 模板

当用户要的是可直接投喂给模型的 Prompt，优先输出如下模板：

```xml
<role>
你是[角色名称]，专注于[领域/问题]。
</role>

<profile>
- 核心价值：[该角色最重要的价值]
- 适用场景：[在哪些情况下使用]
- 默认语言：[中文 / English]
- 默认优先级：[质量 / 速度 / 风险控制 / 证据]
</profile>

<goals>
1. [目标 1]
2. [目标 2]
3. [目标 3]
</goals>

<non_goals>
- [不负责的事项]
- [不应越界处理的事项]
</non_goals>

<background>
[角色所代表的背景、世界观、业务环境、方法论来源]
</background>

<knowledge_boundaries>
- 擅长：[领域范围]
- 不擅长：[越界范围]
- 当证据不足时：[直接说明不知道 / 需要补充数据]
</knowledge_boundaries>

<constraints>
1. 使用结构化输出，不写空泛套话。
2. 不编造事实、经历、数据或来源。
3. 遇到高风险问题时，先说明限制，再给出保守建议。
4. 事实与推断分开表达。
</constraints>

<tools_and_skills>
- 可用工具：[浏览器 / MCP / CLI / 数据库 / 文件处理]
- 可用 skills：[本地已有 skill]
- 如果缺少能力：显式使用 `find-skills` 搜索能提高效率或补齐能力的 skill。
</tools_and_skills>

<workflow>
1. 先澄清任务与目标。
2. 识别已知条件、未知条件与关键假设。
3. 按该角色的方法论给出分析或方案。
4. 在输出前进行一次自检，确认没有越界、没有编造、没有漏掉关键风险。
</workflow>

<output_contract>
- 先给出：[诊断 / 结论 / 判断]
- 再给出：[依据 / 推理 / 方案]
- 最后给出：[行动项 / 风险 / 待补信息]
</output_contract>

<context_management>
- 长对话中定期总结已确认事实、待验证项和当前目标。
- 不要重复读取已经确认的信息。
- 当上下文已过长时，优先压缩为“事实 / 假设 / 决策 / 待办”四段摘要。
</context_management>

<self_check>
- 角色是否保持一致？
- 是否把事实和推断分开？
- 是否越过了知识边界？
- 是否遗漏了风险、限制或前置条件？
</self_check>

<few_shot_examples>
[至少加入 2-3 个高质量示例，覆盖标准场景、边界场景、拒答或补证据场景]
</few_shot_examples>

<initial_message>
我是你的[角色名称]。请提供[任务 / 材料 / 场景]，我会按[该角色方法]给出结构化建议。
</initial_message>
```

## 步骤 5：Few-Shot 不是装饰，必须服务于角色稳定性

默认加入 2-3 个示例，不要只给一个“完美路径”。

最少覆盖：

- 一个标准任务示例
- 一个信息不足示例
- 一个越界或高风险示例

示例要体现的是思考路径和边界控制，不是固定措辞。

不要写成：

- 只有华丽措辞，没有任务差异
- 只展示最终答案，没有展示角色如何约束自己
- 每个示例都同质化

## 步骤 6：显式设计 `find-skills` 补能入口

当角色能力需要外部加成时，不要假装自己已经具备全部能力。直接在结果里提示或调用 `find-skills`。

如果用户口语里写成“find-skill”，一律按 `find-skills` 理解。

以下场景优先考虑 `find-skills`：

- 需要特定文件处理：如 PDF、表格、演示文稿、截图、OCR
- 需要特定研发流程：如测试、代码审查、部署、浏览器自动化
- 需要特定领域方法：如 SEO、定价、用户研究、GitHub 检索
- 需要更强的交付模板：如前端设计、幻灯片、命令生成

输出时要明确写出类似指引：

```markdown
## Recommended Companion Skills
- `find-skills`：当需要补充领域能力、工具链或交付格式时，先用它搜索可安装 skill
- [其他已知 skill]：说明其用途
```

## 步骤 7：加入现实边界，减少幻觉和角色漂移

至少加入这些约束：

- 不知道就说不知道
- 需要证据时明确要求证据
- 不编造人物经历、组织背景、法规条文或实验数据
- 事实与判断分开写
- 高风险领域必须加“仅供参考，建议由专业人士复核”的边界说明

对于“具名人物风格专家”，优先抽取公开可验证的思维模型、工作方法和偏好，不要伪造私人经历、对话和隐私背景。

## 步骤 8：控制 Prompt 密度，保持在“目标清晰、路径灵活”的区间

避免两种极端：

- 太松：只有人设，没有方法、边界和输出契约
- 太死：每一步都写死，导致模型只能机械复述

优先保留这些高价值信息：

- 角色的判断标准
- 角色的边界和拒答规则
- 输出结构
- 工具和 skill 路由
- few-shot 示例

优先删除这些低价值信息：

- 冗长礼貌话术
- 对常识性概念的重复解释
- 不能改变结果质量的背景故事

## 输出要求

使用本 skill 产出结果时，默认按以下顺序输出：

```markdown
# 专家蓝图
## 名称
## 适用场景
## 核心任务
## 非目标
## 方法论与原则
## 边界与风险
## 工具 / Skills 路由

# 通用角色条目
[可直接保存、分享、迁移到任意项目的角色条目]

# 运行时 Prompt
[可直接复制使用的完整 prompt]

# 快速测试提示
1. [标准场景]
2. [边界场景]
3. [高风险或拒答场景]

# 推荐配套 Skills
- `find-skills`：[为什么需要它]
- [其他 skill]：[为什么需要它]
```

## 快速评测清单

交付前至少自检以下问题：

1. 这个角色是否有清晰边界，而不是“万能助手”？
2. 这个角色是否真的体现了领域方法，而不只是换了语气？
3. few-shot 是否覆盖标准、缺信息、越界三类情况？
4. 是否明确告诉模型何时查证、何时拒答、何时补充信息？
5. 是否明确写出 `find-skills` 的使用时机？
6. 输出结构是否能稳定复用？

## 示例

### 示例 1：技术领域专家

输入：

```text
帮我生成一个 Kubernetes 平台架构顾问 agent，要求能做架构评审、故障排查建议和平台演进规划。
```

输出重点：

- 角色边界聚焦平台架构与 SRE，不扩展到泛产品决策
- `Decision Framework` 中体现可靠性、可观测性、成本与演进节奏
- `Tool / Skill Routing` 中明确：如需补充集群运维、浏览器排查、文档抓取能力，可使用 `find-skills`

### 示例 2：业务领域专家

输入：

```text
帮我做一个 B2B SaaS 定价策略顾问角色，要能分析套餐、升级路径和实验方案。
```

输出重点：

- 角色围绕定价与变现，不混入品牌、内容或融资策略
- few-shot 至少覆盖一个“数据不足，先补指标”的场景
- 推荐配套 skill 中显式包含 `find-skills`，用于补充竞争情报、表格分析或调研类能力

## 注意事项

- 优先生成“可执行角色”，不是“文案化角色简介”
- 优先保证 skill 自包含，不要假设存在任何本地模板、目录或参考条目
- 如果用户只是要一个一次性回答，不一定需要创建完整专家 Agent
- 如果任务高度依赖实时信息、法规、数据或外部系统，务必把查证和工具使用写进 Prompt
