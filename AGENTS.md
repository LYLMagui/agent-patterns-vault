# AGENTS.md

本文件是 `agent-patterns-vault` 的仓库级指令，适用于仓库根目录及其所有子目录。

本仓库主要用于个人沉淀、整理和复用以下内容：

- agents
- skills
- MCP
- 提示词模式
- 工作流设计
- 学习笔记
- 从网上收集后经筛选整理的高价值材料

默认目标不是“存得越多越好”，而是“可查找、可复用、可持续整理”。

## 1. 仓库定位

AI 在本仓库中的主要职责是“整理者”和“编目者”，而不是泛化的内容生成器。

优先目标：

1. 把内容放到正确位置
2. 保持条目结构稳定
3. 保留来源和上下文
4. 提高后续检索和复用效率

次要目标：

- 美化文档
- 扩写说明
- 增加示例

如次要目标会影响原始材料真实性、可追溯性或结构稳定性，应优先放弃次要目标。

## 2. 目录职责

### `inbox/`

用于存放刚收集来的原始材料、草稿、待分类内容。

适合放入：

- 从网页、文章、仓库、论坛中摘录的内容
- 尚未验证是否值得长期保留的材料
- 只有片段、没有完整上下文的 prompt / skill / MCP 说明

规则：

- 优先保留来源
- 不强制模板化
- 不应为了“格式统一”而丢失原始信息

### `agents/`

用于存放整理完成、可复用的 agent 条目。

规则：

- 每个 agent 一个独立目录
- 使用 `README.md` 作为入口说明
- 如有原始版本和改造版本，应明确区分

### `skills/`

用于存放整理完成、可复用的 skill 条目。

规则：

- 每个 skill 一个独立目录
- 优先保留触发条件、使用方式、依赖工具

### `mcps/`

用于存放 MCP 相关内容，包括：

- 服务说明
- 配置模板
- 接入步骤
- 示例
- 限制说明

### `patterns/`

用于存放跨项目可复用的设计模式、提示词框架、工作流方案。

### `notes/`

用于存放个人理解、对比分析、踩坑记录和学习总结。

### `catalog/`

用于维护索引和标签。

规则：

- `catalog/` 不放正文
- 正式条目新增或重命名后，应同步更新相关索引

### `templates/`

用于存放标准模板。新增正式条目时优先参考此目录。

## 3. 内容分层规则

本仓库采用“混合型、类型优先”的整理策略。

### 原始层

原始层主要是 `inbox/`，目标是快速入库，避免材料丢失。

### 正式层

正式层包括：

- `agents/`
- `skills/`
- `mcps/`
- `patterns/`

进入正式层前，内容至少应满足以下条件之一：

- 已验证值得复用
- 已补齐关键上下文
- 已能清楚判断类型和用途

如信息不足，不要强行归档到正式层，优先留在 `inbox/` 并补注释说明缺失点。

## 4. AI 整理工作流

AI 在整理本仓库时，默认遵循以下顺序：

1. 先搜索现有内容，避免重复创建
2. 判断内容属于 `inbox` 还是正式层
3. 判断类型：agent / skill / mcp / pattern / note
4. 使用模板补齐最小结构
5. 写入或更新条目
6. 更新 `catalog/` 中的索引或标签
7. 如有必要，在 `notes/` 中补充理解或差异说明

禁止跳过“先搜索现有内容”这一步，避免产生多个近似重复条目。

## 5. 命名规则

- 目录和文件默认使用英文 `kebab-case`
- 正式条目使用语义明确的名字，避免 `misc`、`temp`、`new-skill`、`example2`
- 每个正式条目一个独立目录
- 条目主入口默认为 `README.md`

示例：

- `agents/code-review-orchestrator/`
- `skills/browser-content-fetch/`
- `mcps/github-issues-mcp/`
- `patterns/multi-agent-handoff.md`

## 6. 元信息要求

正式条目建议至少保留以下元信息：

- `title`
- `type`
- `tags`
- `source`
- `status`
- `summary`

如内容来自外部资料，`source` 不得省略。

如 AI 无法确认某项元信息：

- 不得编造
- 应写明 `unknown`、`pending` 或在正文中明确说明待补充

## 7. 状态流转

建议统一使用以下状态：

- `collected`：已收集，尚未整理
- `draft`：已初步整理，但尚未充分验证
- `validated`：已验证，可直接复用
- `archived`：保留存档，不建议继续使用

规则：

- `inbox/` 下内容默认使用 `collected`
- 从 `inbox/` 迁移到正式层时，通常先标记为 `draft`
- 真正经过使用或验证后，再标记为 `validated`

## 8. 来源与改写规则

对于收集来的内容，AI 必须区分：

- 原始内容
- 改写版本
- 摘要版本
- 自己补充的说明

推荐做法：

- 原始材料保留来源链接或出处说明
- 改写内容明确注明“改造自哪里”
- 不要把原文、摘要、改写结果混成一段而不加区分

如果内容来自外部高价值材料，默认优先：

1. 保留来源
2. 写摘要
3. 标注个人理解

不要在没有说明的情况下把外部内容伪装成原创沉淀。

### 英文内容中文化规则

对收集来的 `skill` 和 `agent` 内容，如果正文主要是全英文，AI 应在“尽可能小改动”的前提下，将说明性内容翻译成语义最接近的中文。

要求：

- 优先翻译标题、摘要、章节标题、说明性正文
- 保留代码、命令、路径、参数名、环境变量名、工具名、协议名
- 保留原始结构，避免因为翻译而重写内容组织方式
- 不要把简洁指令扩写成与原文不一致的长篇说明
- 如原文中的术语已有稳定英文用法，可采用“中文 + 英文原词”或直接保留英文术语

默认目标：

- 提高中文检索和理解效率
- 保持与原文语义尽量一致
- 降低因意译过度带来的信息偏移

如某段英文本身就是 prompt、规则文本或执行指令，且翻译后可能影响使用效果，应优先：

1. 保留原文
2. 在附近补中文说明

不要为了中文化而破坏可直接复用性。

### 开源来源标注规则

如果收集来的 `skill`、`agent`、`mcp` 或 `pattern` 明确来自开源项目，AI 在整理成正式条目时，应在条目正文最底部补充来源信息。

来源区块应放在文档底部，建议标题为：

- `## 来源`

至少包含以下信息中的可得部分：

- 项目名
- 仓库地址
- 原始文件路径
- 原始条目名称
- 获取时间或整理时间

推荐格式：

```md
## 来源

- 项目：xxx
- 仓库：https://github.com/xxx/xxx
- 路径：path/to/file
- 条目：original-name
- 备注：基于原始内容做了最小必要整理或中文化
```

规则：

- 来源信息默认放在条目最底部
- 如果无法确认完整来源，不得编造
- 若仅知道仓库或文章地址，至少保留可确认的 URL
- 如果条目是在开源内容基础上二次改造，需明确说明“改造自”而非伪装成原创

### 敏感信息处理规则

AI 在整理 `skill`、`agent`、`mcp` 或相关示例内容时，必须主动检查并处理敏感信息，禁止将真实敏感信息原样写入正式条目。

需要重点检查的内容包括：

- API Key
- Access Token
- Session Key
- Cookie
- 密码
- 私钥、证书、密钥片段
- Webhook 地址中的密钥参数
- 带认证信息的连接串
- 真实手机号、邮箱、身份证明信息
- 真实内网 IP、服务器登录地址、个人目录路径中的隐私信息

处理原则：

- 能替换为占位符的，一律替换为占位符
- 能保留结构但去掉真实值的，优先做脱敏而不是删除整段
- 如果敏感信息对理解流程有帮助，可保留字段名，但必须移除真实值
- 不要把疑似敏感信息直接复制到 `README.md`、模板、索引或正式条目中

推荐占位方式：

- `OPENAI_API_KEY=<YOUR_OPENAI_API_KEY>`
- `token=<REDACTED>`
- `cookie=<REDACTED>`
- `https://example.com/webhook/<REDACTED>`
- `/path/to/your/workspace`

如原始材料中包含真实敏感信息，AI 应：

1. 在整理版本中完成脱敏
2. 避免在正式层重复传播
3. 如确有必要保留原始材料，应优先放在 `inbox/` 并先脱敏后保存

禁止行为：

- 将真实 token、cookie、密码、私钥提交到仓库
- 将用户的真实服务器地址、账号信息、个人标识直接写入模板
- 为了“保留原貌”而无差别保留明显敏感信息

如 AI 无法确定某段内容是否属于敏感信息，应默认按敏感信息处理，先脱敏，再在说明中标注“原文已做脱敏处理”。

## 9. 目录选择规则

如果内容同时像 agent 和 pattern，按以下优先级判断：

1. 可直接执行的具体方案 -> `agents/`
2. 主要是复用技能说明 -> `skills/`
3. 主要是 MCP 集成、配置或服务说明 -> `mcps/`
4. 主要是方法论、结构或流程 -> `patterns/`
5. 主要是个人理解、对比或总结 -> `notes/`

如仍无法确定，先放入 `inbox/`，并在内容顶部写明待判定原因。

## 10. 索引维护规则

新增或重命名正式条目后，应尽量同步更新：

- `catalog/agents-index.md`
- `catalog/skills-index.md`
- `catalog/mcps-index.md`
- `catalog/tags.md`

如果当前任务只允许小范围改动，至少应更新对应类型的索引文件。

## 11. 编辑边界

AI 在本仓库中整理内容时，应遵守以下边界：

- 不要删除未确认无价值的原始材料
- 不要覆盖用户手写的经验总结，除非明确要求重写
- 不要因为模板化而删除原始上下文
- 不要把多个主题强行合并到一个条目
- 不要凭空补充不存在的来源、验证结果或使用效果

允许做的事：

- 规范命名
- 拆分混杂内容
- 补模板结构
- 补标签和索引
- 在不改变原意前提下整理表达

## 12. 新增内容的默认策略

当用户只说“把这段内容存进仓库”但没有给出明确位置时，默认策略如下：

1. 如果内容尚未整理完整，进入 `inbox/`
2. 如果内容已经结构清晰且明显可复用，再进入正式层
3. 若为正式层内容，优先使用 `templates/` 里的对应模板

## 13. 输出语言

本仓库中的说明、总结、标签解释和整理备注，默认使用中文简体。

以下内容可保留原文语言：

- 原始 prompt
- 外部来源摘录
- 英文术语、工具名、目录名

## 14. 长期维护原则

- 先可检索，再追求完美
- 先保留来源，再做抽象提炼
- 先分类归档，再逐步标准化
- 优先沉淀高复用内容
- 保持结构稳定，避免频繁重构目录

## 15. AI 执行提示

当 AI 未来参与本仓库整理时，默认应：

- 先阅读本文件
- 再阅读 `README.md`
- 必要时参考 `CONTRIBUTING.md`
- 创建或整理正式条目时参考 `templates/`

如果任务涉及新增、重组或批量整理内容，应先说明将如何分类，再执行修改。
